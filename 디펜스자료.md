### 전문가용 설명

React 18의 Concurrent Features를 기반으로 컴포넌트 트리를 구성했으며,  
Context API(`AuthContext`, `LanguageContext`, `GameLockContext`)를 통해 전역 상태를 관리합니다.

React Router DOM v6의 `<Routes>` / `<Route>` 기반 SPA 라우팅을 사용해  
인증 상태에 따라 Protected Route와 Guest-only Route를 분리했습니다.

스타일링은 **Tailwind CSS**를 채택했습니다. 과제 기술 요구사항에서 순수 CSS는 불충분하다고 명시했기 때문에, utility-first 방식의 Tailwind로 반응형 디자인을 구현했습니다.

실시간 통신은 **Socket.io-client**로 백엔드 WebSocket 서버와 연결하며,  
게임 세션, 채팅, 친구 온라인 상태 등을 실시간으로 동기화합니다.

다국어 지원은 **i18next**를 통해 구현했으며, `LanguageContext`가 언어 전환 상태를 전역으로 관리합니다.

프로덕션 빌드는 Vite가 번들링하고, **Nginx**가 정적 파일을 서빙합니다.  
별도의 nginx.conf로 프론트엔드 라우팅(SPA fallback)을 처리합니다.

### 어린아이 설명

> 프런트엔드는 **사용자가 눈으로 보는 화면**이에요.

우리 게임을 열면 보이는 로그인 화면, 캐릭터 선택 화면, 게임 화면 — 이것들이 전부 프런트엔드예요.

- **React**: 레고 블록처럼, 화면을 작은 조각(컴포넌트)으로 만들어 조립하는 도구예요.
- **TypeScript**: 코드에 "이건 숫자야, 이건 문자야"라고 미리 표시해두면 실수를 줄여줘요.
- **Tailwind CSS**: 버튼을 파란색으로 만들거나 화면 크기에 따라 레이아웃을 바꿀 때 쓰는 스타일 도구예요. 핸드폰으로 봐도, 컴퓨터로 봐도 잘 보이게 해줘요.
- **Socket.io**: 채팅처럼 서버와 **실시간**으로 대화하는 통로예요. 친구가 접속하면 바로 알 수 있어요.
- **i18next**: 한국어, 영어, 일본어 등 여러 나라 언어로 화면을 바꿔주는 번역 시스템이에요.
- **Vite**: 개발할 때 코드 수정하면 화면을 거의 즉시 새로고침해주는 빠른 빌드 도구예요.

### 프런트엔드 특이점

1. **Protected Route 패턴**: 로그인하지 않은 사용자는 게임 페이지에 접근할 수 없고 자동으로 로그인 화면으로 이동시킵니다.
2. **GameLockContext**: 게임 중 다른 페이지로 이동하는 것을 막는 잠금 상태를 전역으로 관리합니다.
3. **SPA (Single Page Application)**: 페이지 이동 시 전체 새로고침 없이 필요한 부분만 교체합니다. 빠르고 앱처럼 느껴집니다.
4. **30개 이상의 라우트**: 튜토리얼(클래스별 분리), 게임모드(solo/party/AI/offense), 리더보드, 상점 등 풍부한 페이지 구성.

#### 1. Tailwind CSS — 메인 스타일링 도구

**Tailwind CSS가 뭔가요?**

CSS 프레임워크의 두 가지 철학이 있어요:

- **컴포넌트 기반** (Bootstrap, Material-UI): 이미 만들어진 버튼, 카드, 모달을 가져다 씀
- **유틸리티 기반 (Utility-First)** (Tailwind): 작은 단위의 CSS 속성을 클래스로 조합해서 직접 만듦

Tailwind는 후자예요. 예를 들어:

```jsx
// Bootstrap 방식 — 미리 만들어진 btn 클래스를 가져다 씀
<button class="btn btn-primary">로그인</button>

// Tailwind 방식 — 속성을 직접 조합
<button className="bg-yellow-600 hover:bg-yellow-700 text-white font-semibold py-2 px-4 rounded transition-colors">
  로그인
</button>
```

**실제 우리 코드 예시 (`LoginPage.tsx`):**
```jsx
<div className="min-h-[calc(100vh-8rem)] flex items-center justify-center px-4">
  <div className="bg-gray-800 rounded-lg border border-gray-700 p-8">
    <h2 className="text-2xl font-bold text-yellow-400 text-center mb-6">
      로그인
    </h2>
    <input className="w-full bg-gray-700 border border-gray-600 rounded px-3 py-2
                      text-white focus:outline-none focus:border-yellow-500" />
  </div>
</div>
```

각 클래스의 의미:
| 클래스 | CSS 변환 |
|--------|----------|
| `bg-gray-800` | `background-color: #1f2937` |
| `text-yellow-400` | `color: #facc15` |
| `rounded-lg` | `border-radius: 0.5rem` |
| `p-8` | `padding: 2rem` |
| `hover:bg-yellow-700` | `:hover { background-color: #b45309 }` |
| `focus:border-yellow-500` | `:focus { border-color: #eab308 }` |

**커스텀 컴포넌트 클래스 (`index.css`):**

반복적으로 쓰이는 조합은 `@apply`로 묶어서 재사용했습니다:
```css
@layer components {
  .btn-primary {
    @apply bg-yellow-600 hover:bg-yellow-700 text-white font-semibold py-2 px-4 rounded transition-colors;
  }
  .card {
    @apply bg-gray-800 rounded-lg border border-gray-700 p-6;
  }
  .input-field {
    @apply w-full bg-gray-700 border border-gray-600 rounded px-3 py-2 text-white
           focus:outline-none focus:border-yellow-500;
  }
}
```

#### 2. Inline Style — 동적 값 처리

Tailwind 클래스는 빌드 시점에 결정되기 때문에, **런타임에 계산되는 값** (예: HP 퍼센트)은 처리할 수 없어요. 이런 경우에 한해 React의 `style={{}}` 인라인 스타일을 사용했습니다.

```jsx
// HP바: 현재 HP / 최대 HP 비율을 실시간 계산 → Tailwind로 불가
<div
  className="h-full bg-red-500 rounded-full transition-all duration-300"
  style={{ width: `${Math.max(0, (heroHp / heroMaxHp) * 100)}%` }}
/>

// 영웅별 고유 색상: 게임 데이터에서 동적으로 가져옴
<span style={{ color: entry.color }}>{entry.heroName}</span>
```

즉, **고정된 디자인 → Tailwind**, **숫자로 계산되는 동적 값 → inline style** 로 역할을 나눴습니다.

#### 3. Canvas 2D API — 게임 렌더링

게임 화면(영웅, 몬스터, 맵, 스킬 이펙트)은 HTML `<canvas>` 요소에 JavaScript로 직접 그립니다. CSS 스타일링으로는 이 수준의 실시간 그래픽을 처리할 수 없어요.

```tsx
// Renderer.ts — 캔버스 2D 컨텍스트로 매 프레임 직접 그림
const ctx = canvas.getContext('2d');
ctx.clearRect(0, 0, CANVAS_WIDTH, CANVAS_HEIGHT);  // 이전 프레임 지움
ctx.fillRect(x, y, width, height);                  // 사각형 그리기
ctx.drawImage(sprite, x, y);                        // 스프라이트 그리기
```

#### 왜 Tailwind를 선택했나 — 다른 프레임워크와 비교

| | **Tailwind CSS** | **Bootstrap** | **Material-UI** | **Styled Components** |
|---|---|---|---|---|
| 방식 | 유틸리티 클래스 조합 | 미리 만든 컴포넌트 | 구글 디자인 컴포넌트 | JS 안에 CSS 작성 |
| 커스터마이징 | 매우 자유롭다 | 제한적 (테마 오버라이드 필요) | 제한적 (Material 스타일 강제) | 자유롭다 |
| 번들 크기 | 사용한 클래스만 포함 (매우 작음) | 전체 CSS 포함 (큼) | 전체 포함 (큼) | 보통 |
| 학습 곡선 | 클래스 이름 익혀야 함 | 낮음 (바로 쓸 수 있음) | 낮음 | JS + CSS 동시 이해 필요 |
| 디자인 자유도 | 완전히 자유 | Bootstrap 느낌 강함 | Material 느낌 강함 | 자유 |
| 과제 적합성 | 게임 특화 UI → 적합 | 일반 웹앱 느낌 → 부적합 | 기업형 UI 느낌 → 부적합 | 가능하지만 복잡 |

**우리가 Tailwind를 선택한 이유:**
1. **게임 UI 특성**: 우리 프로젝트는 일반적인 웹 폼이 아니라 게임 인터페이스예요. Bootstrap/Material-UI의 정형화된 컴포넌트는 게임 특유의 UI(어두운 배경, 골드 색상, 게임 HUD)와 맞지 않아요.
2. **Vite + React 궁합**: 빌드 시 실제 사용한 클래스만 CSS에 포함(Tree-shaking)되어 최종 번들이 가볍습니다.
3. **반응형 내장**: `md:`, `sm:` 같은 브레이크포인트가 클래스 수준에서 지원되어 별도 미디어쿼리 작성이 불필요합니다.

---

#### 전문가용 한줄 정리

Tailwind CSS는 PostCSS 플러그인으로 동작하며, Vite 빌드 시 `content` 경로의 파일을 스캔해 실제로 사용된 클래스만 최종 CSS로 추출(PurgeCSS)합니다. 동적 값이 필요한 곳은 React의 인라인 스타일로 보완하고, 게임 그래픽은 Canvas 2D API로 분리하여 각 스타일링 도구를 목적에 맞게 사용했습니다.

#### 어린아이 설명

> 옷을 입히는 방법이 세 가지예요.

- **Tailwind**: 미리 만들어진 옷 조각(클래스)을 골라서 조합해요. "노란 버튼, 둥근 모서리, 큰 글씨" 이런 식으로요.
- **Inline Style**: 특별한 경우(HP가 30%라서 바의 너비가 30%여야 할 때)엔 그 순간에 맞는 옷을 즉석에서 만들어요.
- **Canvas**: 게임 화면은 옷이 아니라 직접 그림을 그리는 거예요. 붓으로 캔버스에 영웅, 몬스터를 그리는 것과 같아요.

### 반응형 디자인 (Responsive Design)

과제 기술 요구사항: *"프런트엔드가 명확하고, 반응형이며, 다양한 기기에서 접근성이 좋아야 한다. 데스크톱과 모바일 기기 모두에서 사용 가능해야 한다."*

#### 핵심 기술: Tailwind CSS 브레이크포인트

Tailwind CSS는 화면 너비에 따라 스타일을 다르게 적용하는 **브레이크포인트(breakpoint)** 시스템을 제공합니다.

| 접두사 | 최소 화면 너비 | 대상 기기 |
|--------|--------------|-----------|
| (없음) | 0px ~ | 모든 화면 (기본값, 모바일 우선) |
| `sm:` | 640px ~ | 태블릿 |
| `md:` | 768px ~ | 태블릿 가로 / 소형 데스크톱 |
| `lg:` | 1024px ~ | 데스크톱 |
| `xl:` | 1280px ~ | 대형 데스크톱 |

Tailwind는 **모바일 우선(Mobile First)** 원칙으로 설계됩니다.  
접두사가 없으면 모든 화면에 적용되고, `md:`를 붙이면 768px 이상에서만 적용됩니다.

#### 실제 코드에서의 동작 (`Layout.tsx`)

**데스크톱 메뉴 (`hidden md:flex`)**
```jsx
<div className="hidden md:flex items-center gap-4">
  {/* 13개 nav 링크 */}
</div>
```
- `hidden` → 기본값(모바일): 이 영역 완전히 숨김
- `md:flex` → 768px 이상(데스크톱): `display: flex`로 가로 메뉴 표시

**햄버거 버튼 (`md:hidden`)**
```jsx
<button className="md:hidden ...">
  {/* ☰ 아이콘 */}
</button>
```
- `md:hidden` → 768px 이상: 숨김 (데스크톱에선 필요 없음)
- (접두사 없음) → 모바일: 표시

**모바일 드롭다운 메뉴 (`md:hidden`)**
```jsx
{mobileOpen && (
  <div className="md:hidden flex flex-col gap-3">
    {/* 세로로 나열된 메뉴 항목들 */}
  </div>
)}
```
- 햄버거 버튼을 클릭하면 `mobileOpen` 상태가 `true`로 바뀌며 드롭다운 표시
- 메뉴 항목 클릭 시 `closeMobile()`이 호출되어 자동으로 닫힘
- 데스크톱(768px 이상)에서는 `md:hidden`으로 아예 렌더링되지 않음

**결과적으로 화면 크기별 동작:**

```
모바일 (< 768px)          데스크톱 (≥ 768px)
─────────────────         ──────────────────────────────
[Hero Defense]  [☰]       [Hero Defense] [Play][Tutorial]...[Profile]
                           ↑ 가로 메뉴 13개 항상 표시
☰ 클릭 시:
▼ Play
▼ Tutorial
▼ Lobby
  ... (세로 목록)
```

#### Flexbox와 footer 고정

footer가 항상 화면 하단에 위치하려면 부모 컨테이너가 **Flexbox 세로 정렬**이어야 합니다.

```jsx
// Layout.tsx 최상위 구조
<div className="min-h-screen flex flex-col">   {/* 전체 높이 확보 + 세로 flex */}
  <nav>...</nav>
  <main className="flex-1">...</main>           {/* 남은 공간 전부 차지 */}
  <footer>...</footer>                          {/* 자동으로 맨 아래 */}
</div>
```

- `min-h-screen`: 최소 높이를 뷰포트 전체로 설정
- `flex flex-col`: 자식 요소들을 위→아래 방향으로 배치
- `flex-1` (main): 남은 공간을 전부 차지 → footer를 자연스럽게 아래로 밀어냄

#### 전문가용 설명

Tailwind의 반응형 시스템은 CSS `@media` 쿼리를 내부적으로 생성합니다.  
`md:flex`는 컴파일 시 `@media (min-width: 768px) { display: flex; }`로 변환됩니다.  
개발자가 별도의 CSS 파일 없이 클래스명 조합만으로 반응형 레이아웃을 구현할 수 있는 것은 이 빌드 타임 변환 덕분입니다.  
Vite 빌드 시 실제로 사용된 클래스만 최종 CSS에 포함(purge)되어 번들 크기를 최소화합니다.

#### 어린아이 설명

> 화면 크기가 핸드폰만 하면 메뉴를 다 보여줄 수 없어요. 그래서 **접이식 메뉴(☰)** 를 만들었어요.

- 컴퓨터 화면(768px 이상): 메뉴가 가로로 쭉 펼쳐져요.
- 핸드폰 화면(768px 미만): 메뉴를 숨기고 ☰ 버튼만 보여요. 누르면 메뉴가 아래로 펼쳐지고, 다시 누르거나 항목을 고르면 닫혀요.
- `md:`는 "화면이 768픽셀보다 클 때만 적용해"라는 명령어예요. 마치 옷 사이즈처럼, 화면 크기에 맞는 레이아웃을 자동으로 골라줘요.

### 전문가용 설명

### 인증 보안 — 해시(Hash)와 솔트(Salt)

**파일 위치:** `backend/src/auth/auth.service.ts`

**실제 코드:**

```typescript
// 회원가입 — backend/src/auth/auth.service.ts:30~31
const salt = await bcrypt.genSalt(10);           // 솔트 생성
const passwordHash = await bcrypt.hash(dto.password, salt);  // 해시 생성

// DB에는 passwordHash만 저장, 원본 비밀번호는 저장하지 않음
await this.prisma.user.create({
  data: { email, username, passwordHash }
});

// 로그인 — backend/src/auth/auth.service.ts:57
const isPasswordValid = await bcrypt.compare(dto.password, user.passwordHash);
// 입력한 비밀번호와 저장된 해시를 비교, 원본 복원 없이 검증
```

---

#### 해시(Hash)란?

**전문가용:**  
해시 함수는 임의 길이의 입력값을 **고정 길이의 문자열로 변환**하는 단방향 함수입니다. `bcrypt`는 암호화 목적의 해시 함수로, 연산이 의도적으로 느리게 설계되어 있어 무차별 대입 공격(Brute-force)에 강합니다. 결정적으로 **역방향 복원이 수학적으로 불가능**합니다.

```
"mypassword123"  →  bcrypt  →  "$2b$10$N9qo8uLOickgx2ZMRZo.."
                               ↑ 이 문자열에서 원본을 되돌릴 수 없음
```

**어린아이 설명:**  
비밀번호를 종이에 적어서 창고에 보관하는 게 아니라, **믹서기로 갈아버린 다음** 창고에 보관하는 거예요. 나중에 로그인할 때도 입력한 비밀번호를 같은 믹서기로 갈아서 창고에 있는 것과 비교해요. 원본이 필요 없으니까 설령 창고(DB)가 털려도 비밀번호를 알 수 없어요.

---

#### 솔트(Salt)란?

**전문가용:**  
솔트는 해시 전에 비밀번호에 **덧붙이는 무작위 값**입니다. 같은 비밀번호여도 솔트가 다르면 완전히 다른 해시값이 나옵니다. 이를 통해 두 가지 공격을 방어합니다.

1. **레인보우 테이블 공격 방어**: 공격자가 미리 계산해둔 해시 목록(레인보우 테이블)과 비교하는 공격. 솔트가 추가되면 그 목록이 쓸모없어집니다.
2. **동일 비밀번호 노출 방지**: 여러 유저가 같은 비밀번호를 써도, 솔트가 다르기 때문에 DB에 저장된 해시값이 서로 다릅니다.

```
유저 A: "1234" + saltA → "$2b$10$AAA..."
유저 B: "1234" + saltB → "$2b$10$BBB..."
           ↑ 같은 비밀번호지만 DB에 저장된 값이 완전히 다름
```

`bcrypt.genSalt(10)`의 숫자 `10`은 **cost factor(비용 인수)**입니다. 2^10 = 1,024번의 연산을 거쳐 해시를 생성한다는 의미로, 숫자가 클수록 해시 생성이 느려져 공격자의 시도 횟수를 줄입니다. 현재 `10`은 보안과 성능의 업계 표준값입니다.

**어린아이 설명:**  
솔트는 비밀번호에 **랜덤 양념을 추가**하는 거예요. "1234"라는 비밀번호도 양념이 달라지면 완전히 다른 맛(해시값)이 나와요. 공격자가 "1234를 갈면 이런 결과가 나오더라"라고 미리 계산해뒀어도, 양념 때문에 그 계산이 전부 무쓸모가 돼요.

---

#### 전체 인증 흐름 요약

```
[회원가입]
사용자 입력: "mypassword"
    ↓
bcrypt.genSalt(10)  →  무작위 솔트 생성 (예: "$2b$10$N9qo8...")
    ↓
bcrypt.hash("mypassword", salt)  →  해시값 생성
    ↓
DB 저장: passwordHash = "$2b$10$N9qo8uLOickgx2ZMRZo..."
         (원본 "mypassword"는 어디에도 저장되지 않음)

[로그인]
사용자 입력: "mypassword"
    ↓
DB에서 passwordHash 조회
    ↓
bcrypt.compare("mypassword", passwordHash)
    → 내부적으로 같은 솔트로 다시 해시해서 비교
    → true: 로그인 성공, JWT 발급
    → false: 로그인 실패 (UnauthorizedException)
```

**Rate Limiting:**  
`@nestjs/throttler`로 전역 가드를 적용했습니다.
- `short`: 1초에 최대 20회 요청
- `long`: 1분에 최대 200회 요청  
→ Brute-force 공격 및 DDoS를 방어합니다.

**입력 유효성 검사:**  
`class-validator` + `class-transformer`를 통해 모든 DTO에서 서버 측 유효성 검사를 수행합니다. 잘못된 형식, 빈 값, 범위 초과 등을 자동으로 400 Bad Request로 차단합니다.

---

### 폼 유효성 검사 (Form Validation)

과제 요구사항: *"모든 폼과 사용자 입력값은 프런트엔드와 백엔드 모두에서 유효성 검사를 거쳐야 한다."*

유효성 검사는 **두 겹**으로 구성되어 있습니다.

---

#### 1단계 — 프런트엔드 유효성 검사

**파일 위치:** `frontend/src/pages/RegisterPage.tsx`

HTML5 기본 속성과 JavaScript 검사를 함께 사용합니다.

**HTML5 속성 검사 (`RegisterPage.tsx:75~120`):**
```jsx
<input type="email"    required />           // 이메일 형식 아니면 제출 자체 차단
<input type="text"     required minLength={3} maxLength={20} />  // 길이 제한
<input type="password" required minLength={8} />                 // 최소 8자
```
브라우저가 제출 전에 먼저 막아줍니다. 서버까지 요청이 가지 않아요.

**JavaScript 추가 검사 (`RegisterPage.tsx:21~34`):**
```typescript
if (password !== confirmPassword) {
  setError(t.register.passwordMismatch);   // 비밀번호 확인 불일치
  return;
}
if (password.length < 8) {
  setError(t.register.passwordTooShort);   // 비밀번호 8자 미만
  return;
}
if (username.length < 3 || username.length > 20) {
  setError(t.register.usernameLengthError); // 닉네임 길이 범위 초과
  return;
}
```
이 검사들은 서버 요청 전에 즉시 에러 메시지를 보여줍니다.

---

#### 2단계 — 백엔드 유효성 검사

프론트엔드는 **우회 가능**합니다. 브라우저를 거치지 않고 터미널(curl, Postman 등)로 직접 API를 두드리면 프론트엔드 검사는 완전히 건너뜁니다. 그래서 백엔드 검사가 반드시 필요합니다.

**[1] 전역 ValidationPipe — `backend/src/main.ts:20~24`**

```typescript
app.useGlobalPipes(new ValidationPipe({
  whitelist: true,            // DTO에 없는 필드는 자동으로 제거
  forbidNonWhitelisted: true, // DTO에 없는 필드가 오면 400 에러 반환
  transform: true,            // 타입 자동 변환 (string → number 등)
}));
```

모든 요청이 백엔드에 도착하는 순간 이 파이프를 거칩니다.  
DTO에 정의된 규칙을 통과하지 못하면 **코드 한 줄도 실행되지 않고 즉시 400 Bad Request**를 반환합니다.

**[2] DTO 유효성 규칙 — `backend/src/auth/dto/register.dto.ts`**

```typescript
export class RegisterDto {
  @IsEmail()                 // 이메일 형식이 아니면 차단
  email: string;

  @IsString()
  @MinLength(3)              // 3자 미만 차단
  @MaxLength(20)             // 20자 초과 차단
  @Matches(/^[a-zA-Z0-9_가-힣...]+$/, {
    message: '...'           // 허용 문자 외 특수문자, 공백 차단
  })                         // → XSS 시도 '<script>' 같은 값 차단
  username: string;

  @IsString()
  @MinLength(8)              // 8자 미만 차단
  @MaxLength(100)            // 100자 초과 차단
  password: string;
}
```

**[3] Rate Limiting — `backend/src/app.module.ts:17~20`**

```typescript
ThrottlerModule.forRoot([
  { name: 'short', ttl: 1000,  limit: 20  },  // 1초에 20회 초과 차단
  { name: 'long',  ttl: 60000, limit: 200 },  // 1분에 200회 초과 차단
])
```

같은 IP에서 반복적으로 요청을 쏟아부으면(터미널 공격) 자동으로 `429 Too Many Requests`를 반환합니다.

---

#### 디펜스에서 봤던 "터미널 공격이 막히는 것" — 정확히 이렇게 동작했어요

```bash
# 예: 터미널에서 이상한 값으로 직접 API 요청
curl -X POST https://서버/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"email": "not-an-email", "username": "<script>alert(1)</script>", "password": "123"}'
```

**막히는 과정:**
1. `email: "not-an-email"` → `@IsEmail()` 위반 → **즉시 400**
2. `username: "<script>alert(1)</script>"` → `@Matches()` 정규식 위반 → **즉시 400**
3. `password: "123"` → `@MinLength(8)` 위반 → **즉시 400**
4. 같은 요청을 1초에 20번 이상 → Rate Limiting → **429**
5. DTO에 없는 필드(`"admin": true` 등) → `forbidNonWhitelisted` → **즉시 400**

이 모든 차단은 `auth.service.ts` 코드가 실행되기 **전에** 발생합니다. DB에는 절대 닿지 않아요.

---

#### 두 겹 구조가 필요한 이유

```
사용자 → [프런트엔드 검사] → [백엔드 검사] → DB
           ↑ 편의성 목적       ↑ 보안 목적
           빠른 에러 표시      프론트 우회 방어
           서버 요청 절약      SQL Injection 차단
                              XSS 입력 차단
```

프런트엔드 검사만 있으면: 브라우저 없이 터미널로 바로 뚫립니다.  
백엔드 검사만 있으면: 사용자가 에러를 서버 응답 올 때까지 기다려야 합니다. UX가 나빠요.  
**둘 다 있어야 완전합니다.**

**WebSocket:**  
`@nestjs/websockets` + `socket.io`를 사용해 게임 세션의 실시간 동기화(웨이브 진행, 몬스터 이동, 플레이어 액션), 채팅, 온라인 상태를 처리합니다.

---

### 어린아이 설명

> 백엔드는 **식당의 주방**이에요. 손님은 주방을 직접 볼 수 없지만, 주방에서 음식(데이터)을 만들어서 내보내줘요.

- **NestJS**: 레시피(모듈) 단위로 주방을 정리하는 시스템이에요. Auth 주방, Game 주방, Shop 주방이 각자 맡은 역할만 해요.
- **JWT**: 로그인하면 받는 **출입증** 같은 거예요. 이 출입증이 없으면 게임 데이터에 접근할 수 없어요.
- **bcrypt**: 비밀번호를 뒤죽박죽 섞어서(해시) 저장해요. 설령 데이터가 유출돼도 원래 비밀번호를 알 수 없어요.
- **Socket.io**: 게임 중에 "적이 나타났어!", "웨이브 클리어!" 같은 소식을 **모든 플레이어에게 동시에** 전달해요.
- **Rate Limiting**: 1초에 20번 이상 요청하면 차단해요. 해킹 시도나 무차별 공격을 막아요.
- **class-validator**: 회원가입할 때 이메일 형식이 맞는지, 비밀번호 길이가 맞는지 서버에서도 한번 더 확인해요.

---

### 백엔드 특이점

1. **모듈 분리 아키텍처**: 각 기능이 독립 모듈로 분리되어 있어 유지보수와 확장이 용이합니다.
2. **전역 Rate Limiting**: `APP_GUARD`로 모든 엔드포인트에 자동 적용됩니다. 개별 설정 불필요.
3. **Prisma 글로벌 모듈**: `PrismaModule`을 전역으로 공유하여 모든 모듈에서 DB 접근 가능.
4. **Health Check 엔드포인트**: `/health` 라우트로 서버 상태를 모니터링할 수 있습니다. Docker healthcheck와 연동됩니다.
5. **WebSocket + REST 혼용**: 일반 CRUD는 REST API, 실시간 게임 로직은 WebSocket으로 역할 분리.

### 전문가용 설명

PostgreSQL 16을 채택했습니다. 복잡한 관계형 데이터(영웅-스킬-탤런트 트리, 유저-게임세션-파티)를 처리하는 데 관계형 DB가 적합하며, ACID 트랜잭션 보장이 필요한 재화(gold/crystals) 처리에도 안정적입니다.

**Prisma ORM의 역할:**
- `schema.prisma`에서 타입 안전한 스키마를 정의하고 `prisma migrate`로 마이그레이션을 관리합니다.
- `prisma generate`로 타입스크립트 클라이언트를 자동 생성하여 쿼리 시 타입 오류를 컴파일 시점에 잡습니다.
- Raw SQL 없이도 복잡한 중첩 include 쿼리가 가능합니다.

**주요 설계 결정:**
- `@map` 어노테이션으로 TypeScript camelCase ↔ DB snake_case 변환을 자동화했습니다.
- `@@unique` 복합 유니크 제약으로 중복 친구 요청, 중복 업적 수령 등을 DB 레벨에서 방지합니다.
- `onDelete: Cascade`로 유저 삭제 시 연관 데이터(영웅, 채팅, 업적)가 자동 정리됩니다.
- `Json` 타입 컬럼(`protagonistSave`, `partyJson`, `effectJson`)으로 게임 세이브 데이터처럼 스키마가 유동적인 데이터를 유연하게 저장합니다.
- 자동 백업: 매일 새벽 2시 `pg_dump`로 백업 실행, 7일치 보관 (`backup.sh`).

### DB vs 프론트엔드 — 어디서 무엇을 처리하는가

디펜스에서 자주 나오는 질문: *"캐릭터 능력치, 몬스터 스탯, 전투 계산이 DB에 있나요?"*

**정확한 역할 구분:**

| 데이터/로직 | 위치 | 이유 |
|-------------|------|------|
| 유저 계정, 비밀번호(해시) | DB — `User` 테이블 | 영구 저장 필요 |
| 영웅 기본 능력치 (`baseHp`, `baseAtk`, `baseDef`, `baseSpeed`) | DB — `HeroTemplate` 테이블 | 정적 데이터, 공유 참조 |
| 스킬 정의 (데미지, 쿨다운, 범위) | DB — `Skill` 테이블 | 정적 데이터, 공유 참조 |
| 탤런트 효과 | DB — `TalentNode.effectJson` | 영구 저장 필요 |
| 내 영웅 레벨, 보유 여부 | DB — `UserHero` 테이블 | 유저별 영구 저장 |
| 게임 결과 (최고 웨이브, 클리어) | DB — `GameSession` 테이블 | 게임 종료 후 저장 |
| 친구 목록, 채팅 내역 | DB — `Friendship`, `ChatMessage` | 영구 저장 필요 |
| **몬스터 스탯 (hp, atk, def, speed)** | **프론트엔드 — `WaveData.ts`** | 실시간 처리, DB 불필요 |
| **전투 계산 (데미지 공식, 적중 판정)** | **프론트엔드 — `GameEngine.ts`** | 초당 60프레임, DB로 불가 |
| **시너지 계산, 탤런트 효과 적용** | **프론트엔드 — `SynergySystem.ts`** | 실시간 처리 |

**몬스터가 DB 테이블로 존재하지 않는 이유:**

실시간 게임은 초당 60프레임으로 연산이 돌아요. 몬스터가 이동하고 영웅이 공격할 때마다 데미지를 계산하는데, 이 모든 과정을 매 프레임 DB에 저장하거나 조회하면 수백ms 지연이 발생해 게임이 불가능해집니다.

그래서 **전투 로직은 브라우저(프론트엔드)에서 처리**하고, DB에는 **게임이 끝난 후 결과값**만 저장하는 구조로 설계했습니다.

```
게임 진행 중: 브라우저가 모든 것을 계산 (DB 요청 없음)
     ↓
게임 종료 시: 결과(maxWave, isCleared, 획득 gold)만 DB에 한 번 저장
```

**DB에 저장된 `HeroTemplate.baseAtk`와 실제 전투 atk의 차이:**

DB의 `baseAtk`는 영웅의 기초 공격력이에요. 실제 게임에서의 공격력은 여기에 탤런트 보너스, 시너지 보너스, 강화 보너스가 더해져서 최종 결정됩니다. 이 계산은 전부 `GameEngine.ts`가 담당합니다.

```
최종 atk = baseAtk (DB)
           × 시너지 보너스 (프론트엔드 계산)
           + 탤런트 보너스 (DB의 effectJson 읽어서 프론트엔드 적용)
           × 강화 장비 보너스 (프론트엔드 계산)
```

### 어린아이 설명

> 데이터베이스는 **게임의 창고**예요. 내 캐릭터 정보, 아이템, 친구 목록이 여기 저장돼요.

- **PostgreSQL**: 엑셀 표처럼 행(row)과 열(column)로 데이터를 정리하는 창고예요. 아주 믿음직스럽고 오래된 창고예요.
- **Prisma**: 창고 관리인이에요. 창고에서 물건을 꺼내거나 넣을 때 정해진 양식(스키마)대로만 움직이게 관리해줘요.
- **테이블 관계**: 예를 들어, `User`(유저)가 `UserHero`(내 영웅)를 여러 개 가질 수 있고, `UserHero`는 `HeroTemplate`(영웅 도감)을 참조해요. 마치 포켓몬 도감처럼요.
- **몬스터는 왜 DB에 없냐**: 몬스터는 게임 중에만 잠깐 나타나는 캐릭터예요. 게임이 끝나면 사라지죠. 이런 데이터를 굳이 창고에 저장할 필요가 없어요. 대신 게임 결과(몇 웨이브까지 버텼는지)는 창고에 저장돼요.
- **자동 백업**: 매일 새벽 2시에 창고 전체를 복사해서 보관해요. 데이터가 날아가도 복구할 수 있어요.

### 데이터베이스 특이점

1. **14개 테이블, 명확한 정규화**: 게임 데이터 정의(HeroTemplate, Skill, TalentNode 등)와 유저 인스턴스 데이터(UserHero, UserHeroTalent 등)를 분리하여 중복 없이 설계했습니다.
2. **`Json` 컬럼의 전략적 활용**: 세이브 데이터처럼 구조가 자주 바뀌는 데이터는 Json으로, 관계가 명확한 데이터는 외래키로 처리했습니다.
3. **Healthcheck 연동**: `docker-compose.yml`에서 `pg_isready` 헬스체크를 설정하여 DB가 준비되기 전에 백엔드가 시작되지 않도록 `depends_on: condition: service_healthy`로 순서를 보장합니다.
4. **자동 백업 컨테이너**: 별도의 `backup` 서비스가 cron으로 매일 새벽 2시 `pg_dump`를 실행하고 7일치 백업을 유지합니다.

### 전체 컨테이너 구성도

```
외부 접속
    │
    ▼
[ nginx : 8080(HTTP) / 8443(HTTPS) ]
  ├─ HTTP → HTTPS 자동 리다이렉트 (301)
  ├─ /         → frontend:5173  (React 정적 파일)
  ├─ /api/     → backend:3000   (NestJS REST API)
  ├─ /ws/      → backend:3000   (WebSocket)
  └─ /socket.io/ → backend:3000 (Socket.io 핸드셰이크)
       │
       ├── [ backend : 3000 ]  NestJS
       │       └── depends_on: db (healthy 확인 후 기동)
       │
       ├── [ frontend ]        React (Nginx 정적 서버)
       │
       ├── [ db : 5432 ]       PostgreSQL 16
       │       └── Volume: postgres_data (데이터 영속)
       │
       └── [ backup ]          자동 일일 백업 (cron)
               └── Volume: backup_data (백업 파일 영속)

모든 컨테이너 → app-network (브리지, 외부 직접 접근 차단)
```

---

### 전문가용 설명

**왜 Docker를 쓰는가:**  
"내 컴퓨터에서는 됐는데요"를 없애기 위해서입니다. Docker는 OS, 런타임, 의존성을 컨테이너 이미지에 모두 포함시켜 어떤 환경에서도 동일하게 실행됩니다. 과제 요구사항인 "단일 명령어로 전체 애플리케이션 배포"를 `docker-compose up --build` 한 줄로 충족합니다.

**Nginx 리버스 프록시:**  
클라이언트는 Nginx 하나만 바라봅니다. Nginx가 URL 경로에 따라 요청을 내부 서비스로 분기합니다.
- `location /` → 프론트엔드 (정적 파일)
- `location /api/` → 백엔드 REST API
- `location /socket.io/` + `location /ws/` → WebSocket (HTTP Upgrade 헤더 처리)

HTTP 80포트로 들어오면 무조건 HTTPS 8443으로 `301` 리다이렉트합니다.  
TLS 설정은 `TLSv1.2 / TLSv1.3`만 허용하고 취약한 암호화 알고리즘(`aNULL`, `MD5`)을 차단합니다.

---

### 안전한 연결 (HTTPS) — 전체 구조와 코드 위치

과제 요구사항: *"모든 백엔드 연결에 HTTPS가 사용되어야 한다."*

#### 관련 파일 3곳

| 파일 | 역할 |
|------|------|
| `nginx/nginx.conf` | HTTP → HTTPS 리다이렉트, TLS 설정, 인증서 적용 |
| `nginx/generate-certs.sh` | 자체 서명 SSL 인증서 생성 스크립트 |
| `Makefile` | `make` 실행 시 인증서 자동 생성 후 컨테이너 기동 |

---

#### `nginx/nginx.conf` — HTTP → HTTPS 리다이렉트

```nginx
# nginx/nginx.conf:12~16
server {
    listen 80;          # HTTP 포트 수신
    server_name _;      # 모든 호스트명에 적용
    return 301 https://$host:8443$request_uri;
    # 301: 영구 리다이렉트 — 브라우저가 HTTPS 주소를 기억함
    # $host: 요청한 호스트명 (예: localhost)
    # :8443: HTTPS 포트 명시 (생략하면 443으로 가서 연결 실패)
    # $request_uri: 원래 경로 유지 (/api/login → /api/login 그대로)
}
```

**[수정됨] 포트 명시의 이유:**  
수정 전 코드 `https://$host$request_uri`는 포트를 생략해 기본 443 포트로 리다이렉트됐습니다.  
하지만 우리 HTTPS 포트는 **8443**이기 때문에 연결이 실패했습니다.  
`$host:8443`으로 포트를 명시하여 올바른 HTTPS 주소로 이동하도록 수정했습니다.

```
수정 전: http://localhost:8080 → https://localhost     (443, 연결 실패)
수정 후: http://localhost:8080 → https://localhost:8443 (정상)
```

---

#### `nginx/nginx.conf` — HTTPS 서버 설정

```nginx
# nginx/nginx.conf:19~28
server {
    listen 443 ssl;     # HTTPS 포트, SSL 모드
    server_name _;

    ssl_certificate     /etc/nginx/certs/cert.pem;   # 인증서 파일
    ssl_certificate_key /etc/nginx/certs/key.pem;    # 개인키 파일

    ssl_protocols TLSv1.2 TLSv1.3;       # 구버전 TLS(1.0, 1.1) 차단
    ssl_ciphers HIGH:!aNULL:!MD5;        # 취약한 암호화 방식 차단
    ssl_prefer_server_ciphers on;        # 서버가 암호화 방식 선택권 가짐
    ...
}
```

`TLSv1.2 / TLSv1.3`만 허용합니다. TLS 1.0과 1.1은 보안 취약점이 알려진 구버전이라 차단합니다.

---

#### `nginx/generate-certs.sh` + `Makefile` — 인증서 자동 생성

```bash
# nginx/generate-certs.sh:8~11
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout "$CERT_DIR/key.pem" \
  -out   "$CERT_DIR/cert.pem" \
  -subj "/C=KR/ST=Seoul/L=Seoul/O=42Seoul/CN=localhost"
```

```makefile
# Makefile:77~86 — make 실행 시 자동으로 인증서 생성
certs:
    @if [ ! -f nginx/certs/cert.pem ]; then
        openssl req -x509 ...   # 없을 때만 생성
    fi

up: env certs   # up 타겟이 certs에 의존 → make 한 번으로 인증서+컨테이너 자동 처리
```

`make` 명령어 하나로: ① .env 생성 → ② 인증서 생성 → ③ Docker 컨테이너 기동까지 자동 처리됩니다.

---

#### 자체 서명 인증서(Self-signed)란?

```
공식 인증서: 브라우저 ← 인증기관(CA, 예: Let's Encrypt) ← 서버
자체 서명:   브라우저 ← (CA 없음) ← 서버가 직접 서명
```

- 공식 CA(인증 기관)의 검증을 거치지 않았기 때문에 브라우저가 **"안전하지 않음"** 경고를 표시합니다.
- 단, **암호화(TLS) 자체는 정상 동작**합니다. 데이터는 암호화되어 전송됩니다.
- 개발/평가 환경에서는 "고급 → 계속"으로 접속하면 됩니다.
- Makefile에도 안내가 명시되어 있습니다: `⚠ 처음 접속 시 '안전하지 않음' 경고 → 고급 → 계속`

**전문가용:**  
자체 서명 인증서는 공개키 기반(PKI)에서 신뢰 체계(Chain of Trust)를 거치지 않은 것입니다. 암호화 강도는 동일하지만 브라우저가 서버 신원을 검증할 수 없어 경고를 표시합니다. 프로덕션 환경에서는 Let's Encrypt 등 무료 CA를 사용합니다.

**어린아이 설명:**  
자물쇠(인증서)가 있긴 한데, 그 자물쇠를 내가 직접 만든 거예요. 공인된 자물쇠 가게(CA)에서 만든 게 아니라서 브라우저가 "이거 믿어도 돼요?" 하고 물어보는 거예요. 실제로 잠겨있긴 해요 — 데이터는 암호화됩니다.

**프론트엔드 Multi-stage Build:**  
```dockerfile
# Stage 1: Node.js로 빌드
FROM node:20-alpine AS builder
RUN npm install && npm run build   # dist/ 생성

# Stage 2: Nginx로 정적 파일만 서빙
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
```
빌드 도구(Node.js, npm, TypeScript 컴파일러)는 최종 이미지에 포함되지 않습니다. 이미지 크기를 크게 줄이고 공격 표면을 최소화합니다.

**백엔드 Dockerfile 특이점:**  
bcrypt는 C++ 네이티브 모듈이고, Prisma는 OpenSSL에 의존합니다. Alpine 이미지에는 이것들이 없으므로 `apk add openssl python3 make g++`로 빌드 의존성을 설치한 뒤 `prisma generate`로 타입스크립트 클라이언트를 생성합니다.

**서비스 기동 순서 보장:**  
DB가 완전히 준비되기 전에 백엔드가 연결을 시도하면 에러가 납니다. `docker-compose.yml`에서 아래처럼 설정했습니다:
```yaml
db:
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER}"]
    interval: 10s

backend:
  depends_on:
    db:
      condition: service_healthy   # DB healthy 확인 후에만 기동
```
PostgreSQL이 `pg_isready` 응답을 줄 때까지 백엔드 컨테이너 기동을 대기합니다.

**네트워크 격리 및 포트 보안:**  
모든 컨테이너는 `app-network`라는 내부 브리지 네트워크 안에서만 통신합니다. 외부에 열린 포트는 오직 Nginx의 8080(HTTP)과 8443(HTTPS)뿐입니다.

이 부분에서 초기 설정에 보안 취약점이 있었고, 이를 발견하여 수정했습니다.

**[수정됨] backend / db 포트 외부 노출 문제:**

초기 `docker-compose.yml`에는 아래 설정이 있었습니다:

```yaml
# 수정 전 — 취약한 상태
backend:
  ports:
    - "3000:3000"   # 외부에서 http://서버:3000 직접 접근 가능

db:
  ports:
    - "5432:5432"   # 외부에서 PostgreSQL 직접 접근 가능
```

이 설정의 문제:

| 경로 | 결과 |
|------|------|
| `외부 → https://서버:8443/api/` | Nginx 경유, TLS 암호화 O → 정상 |
| `외부 → http://서버:3000/api/` | Nginx 완전 우회, 평문 HTTP → **HTTPS 요구사항 위반** |
| `외부 → 서버:5432` | DB에 직접 접속 시도 가능 → **데이터 노출 위험** |

과제 요구사항: *"For the backend, HTTPS must be used everywhere."* — 포트 3000이 열려 있으면 이 조건을 코드로는 만족할 수 없습니다.

```yaml
# 수정 후 — backend와 db의 ports 항목 제거
backend:
  # ports 없음: app-network 내부에서만 접근 가능
  networks:
    - app-network

db:
  # ports 없음: app-network 내부에서만 접근 가능
  networks:
    - app-network
```

수정 후 외부에서 접근 가능한 포트:

```
8080  → Nginx (HTTP, 즉시 HTTPS로 301 리다이렉트)
8443  → Nginx (HTTPS, TLS 암호화)
────────────────────────────────
3000  → 차단 (내부 전용)
5432  → 차단 (내부 전용)
```

**볼륨으로 데이터 영속성 보장:**  
컨테이너를 삭제하거나 재시작해도 데이터가 유지됩니다.
- `postgres_data`: DB 데이터 파일
- `backup_data`: 일일 자동 백업 파일 (7일치)

---

### 어린아이 설명

> Docker는 **도시락 통**이에요. 음식(프로그램)을 도시락 통에 담아두면, 어디서 열어도 같은 음식이 나와요.

- **Docker**: 우리 앱을 실행하는 데 필요한 모든 것(프로그램, 설정, 라이브러리)을 하나의 상자에 넣어요. 어떤 컴퓨터에서도 똑같이 작동해요.
- **Docker Compose**: 상자(컨테이너)가 5개인데, `docker-compose up` 명령어 하나로 5개를 동시에 켜줘요. 전원 멀티탭처럼요.
- **Nginx**: 회사 안내데스크 같은 역할이에요. 방문자가 오면 "API 요청이면 백엔드로, 화면 요청이면 프론트엔드로" 안내해줘요. HTTP로 오면 "HTTPS로 오세요"라고 자동으로 안내해줘요.
- **Multi-stage Build**: 케이크 만들 때 오븐(빌드 도구)은 필요하지만, 완성된 케이크(최종 앱)를 배달할 때는 오븐을 같이 보내지 않잖아요. 최종 이미지에는 완성된 코드만 들어가서 가벼워요.
- **Healthcheck**: 주방(DB)이 완전히 준비됐는지 확인한 다음에 웨이터(백엔드)가 손님을 받아요. 준비도 안 됐는데 손님 받으면 안 되니까요.
- **Volume**: 컨테이너를 껐다 켜도 데이터가 사라지지 않아요. 외장 하드처럼 데이터를 컨테이너 밖에 따로 저장해둬요.

---

### Docker 특이점

1. **단일 명령어 완전 배포**: `docker-compose up --build` 또는 `make` 한 줄로 5개 서비스가 올바른 순서로 기동됩니다. 수동 개입이 전혀 필요 없습니다.
2. **Nginx가 HTTPS 종단**: 백엔드/프론트엔드는 HTTP로만 통신하고, HTTPS 처리는 Nginx에서 일괄 담당합니다. 내부 서비스 코드 변경 없이 TLS 설정을 중앙 관리할 수 있습니다.
3. **WebSocket 프록시 설정**: `Upgrade: websocket` 헤더와 `Connection: Upgrade`를 Nginx에서 명시적으로 전달합니다. 이 설정이 없으면 Socket.io 연결이 실패합니다. `proxy_read_timeout 86400`(24시간)으로 게임 중 WebSocket 연결이 끊기지 않도록 했습니다.
4. **Alpine 기반 이미지**: `node:20-alpine`, `nginx:alpine`, `postgres:16-alpine` 모두 경량 Alpine Linux를 사용합니다. 불필요한 패키지가 없어 보안 취약점 노출 면적이 줄어듭니다.
5. **자동 백업 컨테이너 분리**: 백업 로직을 별도 컨테이너(`backup`)로 분리하여 DB 컨테이너에 불필요한 cron 데몬을 넣지 않았습니다. 관심사 분리(Separation of Concerns) 원칙을 인프라 레벨에서도 적용했습니다.
6. **[보안 수정] backend/db 포트 외부 노출 차단**: 초기 설정에서 `backend:3000`과 `db:5432`가 `ports`로 호스트에 직접 노출되어 있었습니다. 이 경우 Nginx와 HTTPS를 완전히 우회한 직접 접속이 가능했습니다. `ports` 항목을 제거하여 두 서비스 모두 `app-network` 내부에서만 접근 가능하도록 수정했습니다. 현재 외부로 열린 포트는 Nginx의 8080/8443뿐입니다.

// 수정 전 — catch의 err 타입이 any (TypeScript 나쁜 관행)

## 7. 게임 구조 — 스킬 시스템 아키텍처

디펜스에서 자주 나오는 질문: *"게임 구조를 설명해주세요. OOP인가요? switch-case인가요?"*

---

### 핵심 답변

> **Handler Map 기반 전략 패턴(Strategy Pattern)을 사용했습니다.**  
> switch-case가 아니며, 각 영웅 스킬이 독립 파일로 완전히 분리되어 있습니다.

---

### 구조 설명

#### 전체 파일 구성

```
frontend/src/game/
├── GameEngine.ts          ← 게임 루프, 전투 시뮬레이션 엔진
├── skills/
│   ├── HeroSkills.ts      ← 전체 핸들러를 하나로 합치는 진입점
│   └── handlers/
│       ├── SsrSkills.ts   ← SSR 영웅 스킬 핸들러
│       ├── ArthurSkills.ts ← 아서 영웅 스킬 핸들러
│       ├── FeldahSkills.ts ← 펠다 영웅 스킬 핸들러
│       └── ... (약 100개 영웅 파일)
└── WaveData.ts            ← 몬스터 웨이브 데이터 정의
```

#### 스킬 실행 흐름

```
GameEngine (매 프레임)
    ↓
processHeroSkills(engine, hero, dt, targets)
    ↓
mergedHandlers[skillId]  ← skillId로 딕셔너리에서 핸들러 바로 lookup
    ↓
handler({ engine, hero, dt, validTargets, healMult, skillId })
    ↓
각 영웅 스킬 파일의 함수 실행 (예: SsrSkills.ts, ArthurSkills.ts ...)
```

#### 실제 코드 (`HeroSkills.ts`)

```typescript
// 타입 정의 — 모든 스킬 핸들러의 공통 인터페이스
export type SkillContext = {
  engine: GameEngine;
  hero: GameHero;
  dt: number;
  validTargets: GameMonster[];
  healMult: number;
  skillId: string;
};
export type SkillHandler = (ctx: SkillContext) => boolean;

// 각 영웅 파일의 핸들러를 하나의 딕셔너리로 병합
const mergedHandlers: Record<string, SkillHandler> = {
  ...ssrHandlers,      // SsrSkills.ts
  ...arthurHandlers,   // ArthurSkills.ts
  ...feldahHandlers,   // FeldahSkills.ts
  // ... 약 100개 영웅 핸들러
};

// 실행 — skillId 하나로 바로 찾아서 호출
const handler = mergedHandlers[baseSkillId];
if (handler) {
  const success = handler({ engine, hero, dt, validTargets, healMult, skillId });
}
```

---

### switch-case와 비교

| | **switch-case (안 씀)** | **우리 Handler Map** |
|---|---|---|
| 새 영웅 추가 | GameEngine 코드 수정 필요 | 새 파일만 추가 |
| 영웅 로직 위치 | GameEngine.ts 안에 전부 | 각 영웅 파일에 분리 |
| 코드 규모 | 하나의 거대한 switch | 파일당 독립적 |
| 런타임 탐색 | 위에서 순서대로 비교 | 딕셔너리 O(1) 직접 접근 |

switch-case였다면 `GameEngine.ts` 안에 이런 코드가 있었을 거예요:
```typescript
// ← 우리 코드에 이런 구조 없음
switch(hero.id) {
  case 'ssr': ssr_attack(); break;
  case 'arthur': arthur_attack(); break;
  case 'feldah': feldah_attack(); break;
  // ... 100개 case
}
```

---

### 완전한 OOP와의 차이

완전한 OOP라면 클래스 상속 구조를 썼을 것입니다:
```typescript
// OOP 방식 (우리가 선택하지 않은 것)
abstract class Hero {
  abstract executeSkill(): void;
}
class Ssr extends Hero {
  executeSkill() { /* SSR 스킬 */ }
}
class Arthur extends Hero {
  executeSkill() { /* 아서 스킬 */ }
}
```

우리는 클래스 상속 대신 **함수형 핸들러 + 딕셔너리 lookup** 방식을 선택했습니다.

**이유:**
- React + TypeScript 환경에서 함수형 방식이 더 자연스러움
- 클래스 인스턴스 생성 없이 순수 함수로 스킬 로직을 작성 가능
- 게임 상태(`GameEngine`)를 매 호출마다 Context로 주입하므로, 영웅 객체가 상태를 직접 보유하지 않아도 됨

---

### 전문가용 한줄 정리

`SkillHandler` 타입을 공통 인터페이스로 정의하고, 각 영웅 스킬 파일이 이 인터페이스를 구현한 함수들의 `Record<skillId, SkillHandler>` 맵을 export합니다. `HeroSkills.ts`는 이 맵들을 스프레드 연산자로 병합하여 단일 `mergedHandlers`를 구성하고, 런타임에 O(1)로 스킬 핸들러를 dispatch합니다. 이는 전략 패턴(Strategy Pattern)의 함수형 구현입니다.

### 어린아이 설명

> 100개의 영웅이 있는데, 각 영웅 스킬을 전화번호부처럼 정리했어요.

- "ssr_slash" → SSR 영웅 공격 함수
- "arthur_shield" → 아서 방어 함수
- "feldah_heal" → 펠다 치유 함수

게임이 돌아갈 때 영웅의 스킬 ID를 전화번호부에서 찾아서 바로 실행해요. 책을 처음부터 끝까지 뒤지는 게(switch-case) 아니라, 이름 보고 바로 페이지를 펼치는(딕셔너리 lookup) 방식이에요.

### 여러 컴퓨터에서 접속하는 방법 (42 클러스터 기준)

**조건**: 서버 PC와 클라이언트 PC가 **같은 네트워크(LAN)** 에 있어야 합니다.  
42 클러스터 내 컴퓨터들은 모두 같은 사설망에 있으므로 조건을 만족합니다.

**단계:**

```
① 서버 역할을 맡을 PC에서 Docker 실행
   $ make

② 서버 PC의 로컬 IP 확인
   $ ip addr show | grep 'inet '
   또는
   $ hostname -I
   → 예: 10.12.34.56

③ 다른 PC 브라우저에서 접속
   https://10.12.34.56:8443

④ "안전하지 않음" 경고 → 고급 → 계속
   (자체 서명 인증서 특성상 정상 현상, 암호화는 동작함)

⑤ 각자 회원가입 → 로그인 → Lobby 메뉴 → 게임 시작
```

### 기술적 동작 원리

#### 왜 가능한가

```
다른 PC 브라우저
    │
    │  https://서버IP:8443
    ▼
[ Nginx — 0.0.0.0:8443 수신 ]   ← 모든 IP에서 들어오는 연결 수락
    ├─ /          → 프론트엔드 React 앱 전송
    ├─ /api/      → NestJS REST API
    └─ /socket.io/ → WebSocket 게임 세션
```

Docker의 포트 바인딩 `8443:443`은 기본적으로 `0.0.0.0`(모든 네트워크 인터페이스)에 바인딩되므로, 서버 PC의 IP 주소로 들어오는 연결을 모두 수락합니다.

#### Socket.io URL이 자동으로 맞춰지는 이유

```typescript
// frontend/src/api/socket.ts
const WS_URL = import.meta.env.VITE_WS_URL || '';  // 빈 문자열
socket = io(`${WS_URL}/ws`, ...);
// → io('/ws', ...) → 현재 접속한 origin에 자동으로 연결
```

브라우저가 `https://10.12.34.56:8443`으로 접속했다면, Socket.io도 자동으로 `wss://10.12.34.56:8443/ws`로 연결합니다. IP 주소를 코드에 하드코딩할 필요가 없습니다.

#### CORS 허용 범위

```typescript
// backend/src/main.ts & game/game.gateway.ts — 동일 정책 적용
허용: localhost (모든 포트)
허용: 10.x.x.x     ← 42 클러스터 사설망 (클래스 A)
허용: 172.16–31.x.x ← 클래스 B 사설망
허용: 192.168.x.x  ← 가정/사무실 공유기 (클래스 C)
차단: 그 외 (인터넷 외부)
```

사설망(RFC 1918)에 해당하는 IP만 허용하므로, 외부 인터넷에서의 무단 접근은 여전히 CORS로 차단됩니다.

### 실시간 게임 동기화 구조 (WebSocket)

멀티플레이 게임 중 화면이 실시간으로 동기화되는 원리:

```
[호스트 PC — 게임 엔진 실행]
  GameEngine.ts가 매 프레임(60fps) 게임 상태 계산
      │
      │  socket.emit('game:state', { state, damageNumbers })
      ▼
[백엔드 — game.gateway.ts]
  호스트 소켓 룸에 state 브로드캐스트
      │
      │  socket.emit('game:state', ...)
      ▼
[게스트 PC — GamePage.tsx]
  handleRemoteState() → setGameState() → Canvas 렌더링
```

**핵심**: 게임 연산은 호스트 PC 브라우저에서만 실행됩니다. 게스트는 호스트가 보내주는 상태를 받아서 화면만 그립니다. 네트워크 지연이 있어도 호스트의 계산 결과는 정확합니다.

### 주의사항

| 상황 | 내용 |
|------|------|
| SSL 경고 | 자체 서명 인증서이므로 다른 PC에서도 "안전하지 않음" 경고 발생 → "고급 → 계속" 클릭 |
| 방화벽 | 서버 PC의 OS 방화벽에서 8443 포트 인바운드가 허용되어야 함 (42 클러스터는 대부분 허용됨) |
| 서버 IP | IP는 재부팅마다 바뀔 수 있으므로 게임 시작 전 `ip addr` 또는 `hostname -I`로 확인 |
| 다른 네트워크 | 인터넷(다른 건물, 외부 네트워크)에서의 접속은 지원하지 않음 |

### 전문가용 설명

멀티플레이 아키텍처는 **호스트-클라이언트(Host-Client) 모델**입니다.  
호스트 브라우저가 게임 시뮬레이션 서버 역할을 하고, 서버(NestJS/Socket.io)는 상태를 중계(relay)합니다.  
게스트는 `game:state` 이벤트를 구독하여 호스트의 `GameState` 스냅샷을 매 틱 수신하고, 로컬 Canvas에 렌더링합니다.  
이 구조는 권위 서버(Authoritative Server) 패턴의 변형으로, 호스트가 권위를 가지며 게스트는 read-only 뷰어로 동작합니다.

CORS는 HTTP 요청(`main.ts`)과 WebSocket 핸드셰이크(`game.gateway.ts`) 양쪽에 각각 설정됩니다.  
RFC 1918 사설망 대역(`10/8`, `172.16/12`, `192.168/16`)만 허용하는 정규식으로, 동일 네트워크 접근은 허용하되 외부 인터넷은 차단합니다.

### 어린아이 설명

> 게임을 같이 하려면 같은 와이파이에 연결된 것처럼 같은 네트워크에 있어야 해요.

- 서버 역할을 하는 PC가 하나 있어요. 이 PC가 게임을 실제로 계산해요.
- 다른 PC들은 그 서버 PC의 주소(`https://IP주소:8443`)로 접속하면 돼요.
- 서버 PC는 게임 상태를 1초에 60번씩 다른 PC에 보내줘요.
- 다른 PC들은 그 정보를 받아서 화면에 그리기만 해요.
- 마치 TV 중계처럼, 진짜 경기는 한 곳에서 열리고 다른 PC들은 중계를 보는 거예요.
